# Image Prompt Pack: smolagents vs LangGraph

## 设计原则

- 所有图片为**扁平化技术插图**风格，统一配色方案
- 主色调：深蓝 `#1e3a5f` + 青色 `#00d4aa` + 暖橙 `#ff6b35` + 中性灰 `#6b7280`
- 背景：纯白 `#ffffff` 或极浅灰 `#f8fafc`
- 字体：无衬线字体（如 Inter / SF Pro），中文使用思源黑体或苹方
- 比例：文章内插图统一 16:9；封面/横幅 2:1 或 21:9

**插入位置**：§2 定义与边界 之后，作为全文核心隐喻的视觉锚点

**用途**：建立"控制流显式光谱"这一核心概念，让读者在文章早期获得视觉框架

**画面构图**：
- 水平光谱条，从左到右渐变
- 左端标注"黑盒"（深色/阴影），右端标注"白盒"（明亮/透明）
- 光谱条上方标注两个框架 logo/名称：smolagents 在左端，LangGraph 在右端
- 中间留白标注"灰盒地带（CrewAI / AutoGen / ...）"，暗示后续文章
- 光谱条下方配 3 个关键词：左端"快速上手"、中间"平衡"、右端"精确控制"

**关键元素**：
- 水平渐变条（深蓝 → 浅蓝 → 白）
- 两个框架的抽象图标（smolagents 用极简方块，LangGraph 用节点连接图形）
- 箭头指示"控制流显式程度"从左到右递增
- 底部小字标注维度名称

**风格**：
- 扁平化信息图（flat infographic）
- 极简线条，无 3D 效果

- 机器人/大脑/科幻元素（避免具象化 AI）
- 文字过小（确保在移动端可读）

## 图 2：smolagents 架构——黑盒 ReAct Loop（内部执行流程）

**插入位置**：§3.2 CodeAgent：代码即 Action 之后

**用途**：展示 smolagents 的内部封装——开发者只看见输入和输出，loop 细节不可见

**画面构图**：
- 上下分层结构
- **上层**：用户输入框（"任务描述"）→ 箭头 → **黑盒区域**
- **黑盒区域**：半透明灰色框，内部隐约可见循环箭头（ReAct: Reason → Act → Observe）
- **黑盒内部细节**（半透明显示，暗示"不可直接访问"）：
  - Memory（上下文窗口）
  - LLM 生成 Python 代码
  - Python 解释器执行
  - Tool 调用结果回传
- **下层**：输出结果框（"执行结果 / 最终答案"）
- 黑盒区域上下各有一个"接口"——上方是 `agent.run()` 调用，下方是返回值

**关键元素**：
- 黑盒用半透明深灰色（`rgba(30, 58, 95, 0.15)`）表示"封装"
- 内部循环用虚线箭头
- 标注 "agents.py ~1,800 行" 在黑盒右下角（小字）
- 安全沙箱标注在黑盒外部（E2B / Docker / WASM）

**风格**：
- 扁平化流程图
- 黑盒半透明效果（CSS `backdrop-filter` 风格）
- 单色+点缀色（深蓝为主，青色高亮接口）

**比例**：16:9

**避免项**：
- 将黑盒画成完全不透明（需要暗示内部有内容但不可直接访问）
- 过多技术细节（这是概念图，不是源码注释）
- 代码片段直接写在图上

## 图 3：LangGraph 架构——Pregel 显式状态机（控制流图）

**插入位置**：§4.1 Pregel：状态机即控制流 之后

**用途**：展示 LangGraph 的显式控制流——开发者精确建模每一步状态流转

**画面构图**：
- 有向图结构，3-4 个节点 + 状态通道
- **节点**（圆角矩形，不同颜色区分类型）：
  - 起点节点（Start）：浅色
  - LLM 节点（LLM Call）：深蓝
  - 工具节点（Tool Node）：青色
  - 条件分支节点（Condition）：菱形，暖橙色
  - 终点节点（End）：浅色
- **边**（箭头连线）：
  - 实线箭头：确定流转
  - 虚线箭头：条件流转（带标注如 "if need_tool"）
- **状态通道**（State Channels）：节点之间流动的数据管道，标注 `state = {"messages": [...], "next": "..."}`
- **Checkpoint 标记**：每个节点下方有小圆点表示"可持久化点"
- **Interrupt 标记**：在某一节点旁标注 ⏸ 暂停图标

**关键元素**：
- 节点内用极简图标（齿轮代表 tool，对话框代表 LLM，菱形代表判断）
- 状态通道用浅色管道表示
- 底部标注 "Pregel Execution Engine"
- 右上角小字："开发者显式定义每一步"

**风格**：
- 扁平化技术流程图
- 类似 DAG（有向无环图）可视化工具风格
- 节点有轻微圆角（8px），无边框阴影

**比例**：16:9

**避免项**：
- 节点画得过于复杂（每个节点只放 1-2 个词）
- 箭头交叉过多（保持图的可读性）
- 3D 效果或阴影

## 图 4：双栏对照——控制流显式程度对比（并排对比图）

**插入位置**：§5.1 同维度对比表 之后

**用途**：将 §5.1 的表格数据转化为直观视觉对比，降低阅读认知负荷

**画面构图**：
- 左右双栏，中间用垂直分隔线
- **左栏标题**：smolagents（黑盒）— 深色顶部条
- **右栏标题**：LangGraph（白盒）— 浅色顶部条
- 每栏包含 3 组对比维度（每组一个图标 + 文字）：
  - 控制流：左"隐藏循环" vs 右"显式图结构"
  - 持久化：左"无"（灰色叉）vs 右"原生 Checkpoint"（绿色勾）
  - 人机交互：左"无" vs 右"原生 Interrupt"
  - 上手难度：左"低"（1-2 星）vs 右"高"（4-5 星）
  - 代码体积：左"小 ~7MB" vs 右"大 ~518MB"
  - 灵活性：左"受限" vs 右"完全可控"
- 底部汇总条：左"快速原型" vs 右"生产编排"

**关键元素**：
- 图标系统统一（勾/叉/星/箭头）
- 颜色编码：smolagents 侧用冷色调（蓝/灰），LangGraph 侧用暖色调（橙/青）
- 分隔线不是简单直线，而是"光谱渐变条"的垂直版本

**风格**：
- 扁平化对照卡片
- 类似产品对比页设计（如 Vercel vs Netlify 对比图）
- 高对比度，确保打印时也可读

**比例**：16:9

**避免项**：
- 信息过载（只选 5-6 个最关键维度）
- 使用真实 logo（避免版权风险，用抽象图形代替）
- 文字过小

## 图 5：失败模式雷达图——两个框架的风险维度

**插入位置**：§6 失败模式深度分析 之后

**用途**：将两个框架的弱点可视化，帮助读者同时理解"两个框架都不完美"

**画面构图**：
- 双雷达图叠加（或左右并排的两个雷达图）
- 6 个维度（雷达图的 6 个轴）：
  1. 安全风险
  2. 学习曲线
  3. 维护可持续性
  4. 版本碎片化
  5. 过度工程化风险
  6. 生态锁定风险
- **smolagents 雷达**：在安全风险和可持续性上"面积"大（弱点突出），在学习曲线上"面积"小
- **LangGraph 雷达**：在学习曲线和版本碎片化上"面积"大，在可持续性上"面积"小
- 两个雷达用不同颜色半透明叠加，重叠区域显示"两者共有的风险"

**关键元素**：
- 6 轴标注清晰
- 填充区域用半透明色
- 中心标注 "低风险"，外围标注 "高风险"
- 图例：深蓝 = smolagents，青色 = LangGraph

**风格**：
- 扁平化数据可视化
- 类似 Grafana 或 Datadog 监控面板风格
- 简洁网格线，无背景装饰

**比例**：16:9

**避免项**：
- 雷达图超过 6-7 个维度（会变得混乱）
- 填充色不透明（必须半透明以显示叠加）
- 数值标签直接写在图上（用图例代替）

## 图 6：决策树——如何选择你的 Agent 框架

**插入位置**：§7.1 决策矩阵 之后

**用途**：将 §7.1 的决策矩阵转化为可跟随的流程图，降低决策认知负荷

**画面构图**：
- 树状流程图，从上到下
- **根节点**："开始选择 Agent 框架"
- **第一层分支**（菱形判断）："需要状态持久化/故障恢复？"
  - 是 → 指向 LangGraph
  - 否 → 继续
- **第二层分支**："需要 human-in-the-loop？"
  - 是 → 指向 LangGraph
  - 否 → 继续
- **第三层分支**："需要精确控制每一步执行？"
  - 是 → 指向 LangGraph
  - 否 → 继续
- **第四层分支**："快速原型 / 多模态输入 / 预算敏感？"
  - 是 → 指向 smolagents
  - 否 → "重新评估：也许不需要框架"
- **终端节点**：
  - LangGraph 终端：标注 "精确控制 + 持久化 + 可观测性"
  - smolagents 终端：标注 "快速上手 + 轻量 + 多模态"
  - "不需要框架"终端：标注 "单步 LLM 调用可能已足够"

**关键元素**：
- 判断节点用菱形（标准流程图）
- 终端节点用圆角矩形
- "不需要框架"终端用暖橙色高亮（呼应 Anthropic "简单 > 复杂"）
- 箭头旁标注"是/否"

**风格**：
- 扁平化流程图
- 类似 Mermaid / Draw.io 风格但更加精致
- 路径用颜色区分：蓝色路径 → smolagents，青色路径 → LangGraph，橙色路径 → 不需要框架

**比例**：16:9（或 2:1 如果树太宽）

**避免项**：
- 分支过多（控制在 4 层以内）
- 路径交叉（保持树状清晰）
- 终端节点没有明确标注
