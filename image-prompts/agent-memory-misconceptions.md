# Image Prompt Pack: 记忆幻象 — Agent 框架的「记忆」承诺与工程现实

## 设计原则

- 所有图片为**扁平化技术插图**风格，统一配色方案
- 主色调：深蓝 `#1e3a5f` + 青色 `#00d4aa` + 暖橙 `#ff6b35` + 中性灰 `#6b7280`
- 背景：纯白 `#ffffff` 或极浅灰 `#f8fafc`
- 字体：无衬线字体（如 Inter / SF Pro），中文使用思源黑体或苹方
- 比例：文章内插图统一 16:9；封面/横幅 2:1 或 21:9

## 图 1：「memory」一词在七个框架中的语义图谱

**插入位置**：§1 开头钩子 之后

**用途**：建立全文核心隐喻——同一个词在不同框架中指向完全不同的技术实现，制造认知失调的视觉冲击

**画面构图**：
- 中心大字 "MEMORY"，周围放射状分布七个框架的抽象节点
- 每个节点用不同颜色/形状区分其「记忆」的语义类型：
  - 深蓝色节点：LangGraph（checkpoint / state channels）——标注 "durable execution"
  - 青色节点：Mem0（向量数据库 / ADD-only）——标注 "external layer"
  - 暖橙色节点：Letta（Block / in-context）——标注 "core abstraction"
  - 灰色节点：CrewAI（memory/knowledge/rag 三合一）——标注 "unified storage"
  - 浅蓝节点：AutoGen（ListMemory/Redis/Mem0/Chroma 多后端）——标注 "no unified abstraction"
  - 暗灰节点：OpenHands（EventStream）——标注 "execution trace"
  - 白色虚线节点：smolagents（无内置）——标注 "none"
- 节点之间的连线用不同线型表示语义距离：
  - 实线：语义相近（如 Mem0 和 CrewAI memory 都基于向量存储）
  - 虚线：语义中等关联
  - 无连线：语义无关（如 LangGraph checkpoint 和 Letta Block）

**关键元素**：
- 中心 "MEMORY" 用深灰色，带微妙的光晕暗示「概念泛化」
- 七个节点均匀分布在圆周上，大小与项目 star 数成正比（视觉暗示社区规模）
- 底部图例：实线 = 语义相近，虚线 = 中等关联，无线 = 语义断裂
- 右下角小字标注 "7 frameworks, 7 incompatible implementations"

**风格**：
- 扁平化概念图（flat concept map）
- 类似学术论文中的概念关系图
- 节点用圆角矩形，非圆形（更现代）

**比例**：16:9

**避免项**：
- 真实框架 logo（版权风险）
- 过多连线导致图混乱（最多 10-12 条连线）
- 3D 效果或阴影

## 图 2：记忆分层模型——从上下文缓存到可审计知识库

**插入位置**：§2 误区一（上下文窗口 ≠ 记忆）之后

**用途**：建立「记忆不是单一概念，而是一个从瞬时到持久的频谱」这一核心框架

**画面构图**：
- 垂直四层堆叠结构，从上到下依次是：
  - **第一层（顶部，浅灰色）**：上下文缓存（Context Cache）——标注 "128K~10M tokens | 当前 session | 无结构"
  - **第二层（浅蓝色）**：工作记忆（Working Memory / Letta Block）——标注 "reserved section | Agent 可编辑 | in-context"
  - **第三层（青色）**：语义记忆层（Semantic Memory / Mem0）——标注 "vector store | multi-signal retrieval | cross-session"
  - **第四层（底部，深蓝色）**：可审计知识库（Auditable Knowledge Base）——标注 "versioned history | entity-linked | governance"
- 每层之间有向下箭头，标注数据流动方向
- 每层右侧标注「更新触发」：
  - 第一层：每轮 LLM 调用自动
  - 第二层：Agent 显式工具调用
  - 第三层：LLM ADD-only 提取
  - 第四层：人工审核 + 自动策略

**关键元素**：
- 每层用不同深度颜色表示「持久性」递增（浅 → 深）
- 每层高度与「容量」成反比（上下文缓存最高但最浅，知识库最矮但最深）
- 左侧垂直轴标注 "Persistence"，箭头向上递增
- 底部标注 "记忆的核心挑战不是容量，而是结构与更新策略"

**风格**：
- 扁平化分层信息图
- 类似数据库架构图或缓存层级图
- 每层用细微的边框区分

**比例**：16:9

**避免项**：
- 大脑/海马体等生物记忆隐喻（本文讨论工程实现，不是神经科学）
- 各层之间画成管道（太具象），用简洁箭头即可
- 文字过多导致层内拥挤

## 图 3：CrewAI memory/knowledge/rag 三子系统的功能重叠区域

**插入位置**：§3 误区二（RAG ≠ 长期记忆）之后

**用途**：用维恩图直观展示三个子系统的宣称边界与实际重叠，暴露概念混乱

**画面构图**：
- 三个相交的圆，分别标注：
  - **左圆（蓝色）**：memory —— "对话历史 | 用户偏好 | 自动提取"
  - **右圆（青色）**：knowledge —— "文件源 | PDF/CSV/Excel | 手动加载"
  - **下圆（灰色）**：rag —— "多 provider embedding | 检索增强"
- **三重交集区域（中心，深色）**：用放大插图展示内部细节——标注 "ChromaDB / LanceDB + embedding retrieval"（实际共享底层）
- **非重叠区域**用浅色高亮，标注真正的差异化：
  - memory 独有：对话历史特化提取
  - knowledge 独有：多格式文件解析
  - rag 独有：20+ embedding provider 路由
- 三个圆的交叠程度反映实际代码重叠（约 60-70% 共享底层）

**关键元素**：
- 中心交集区域用深色 + 斜线纹理，暗示「这里才是真相」
- 每个圆的边缘标注 CrewAI 源码中的对应目录（`memory/`, `knowledge/`, `rag/`）
- 底部对比条：宣称的 "3 independent subsystems" vs 实际的 "1 vector store + 3 API wrappers"
- 右下角标注来源 "crewAI v1.14.3"

**风格**：
- 扁平化维恩图
- 类似数据隐私政策中的「数据使用范围」可视化
- 重叠区域用纹理区分（斜线或点阵），避免仅靠颜色区分（色盲友好）

**比例**：16:9

**避免项**：
- 三个圆画得完全重叠（需要保留非重叠区域以展示真正的差异化）
- 中心区域不写任何文字（必须明确标注共享底层是什么）
- 使用贬义视觉语言（这是客观分析，不是攻击）

**用途**：用对比图直观展示「同一个技术机制被包装成两个不同的用户价值承诺」

**画面构图**：

- **左栏标题**：Checkpoint —— 深蓝顶部条
  - 核心图标：一个摄像机/快照图标（snapshot）
  - 功能列表：
    - "失败后从快照恢复"
    - "执行状态序列化"
    - "langgraph-checkpoint v4.0.3"
  - 时间轴：横向时间线，标注 "step 1 → step 2 → [crash] → restore from checkpoint → step 3"
  - 底部标签："Durable Execution"（执行恢复）
- **右栏标题**：Learnable Memory —— 青色顶部条
  - 核心图标：一个生长/学习图标（向上箭头 + 节点）
  - 功能列表：
    - "跨 session 积累知识"
    - "调整未来行为"
    - "用户画像演进"

**关键元素**：

- 右栏用暖色调（青/橙），暗示「用户价值」
- 时间轴用不同方向（横向 vs 纵向）暗示两种不同的时间观（单 session 恢复 vs 跨 session 演进）
- 底部小字："LangGraph README 将二者并列，但实现上 checkpoint 只解决左栏问题"

**风格**：
- 扁平化双栏对比图
- 类似 Stripe 文档中的「before/after」对比风格
- 高对比度，确保打印可读

**比例**：16:9

- 暗示 LangGraph "欺骗" 用户（本文分析的是概念借用现象，不是道德判断）
- 左栏画得过于技术化、右栏画得过于抽象（保持视觉平衡）
- 时间轴上没有具体步骤标注

## 图 5：Mem0（外部记忆层）vs Letta（in-context Block）架构对比

**插入位置**：§5 独立记忆层的崛起 之后

**用途**：展示两条互补道路的根本架构差异——记忆在外部 vs 记忆在上下文

**画面构图**：
- 左右双栏，中间用双向箭头标注 "互补而非竞争"
- **左栏标题**：Mem0 —— 外部独立层 —— 蓝色
  - 顶部：Agent 运行时（抽象方块）
  - 中间：API 调用箭头（虚线，标注 `memory.search()` / `memory.add()`）
  - 下方：Mem0 服务层（圆角矩形，内部分三路：Semantic / BM25 / Entity）
  - 底部：向量数据库（圆柱形图标，标注 "15+ backends"）
  - 右侧特性标签："框架无关" "ADD-only" "多信号检索"
- **右栏标题**：Letta —— 核心架构抽象 —— 青色
  - 顶部：Agent 运行时（抽象方块，内部标注 "Context Window"）
  - 中间：四个水平层叠的 Block（Core / Recall / Archival / Summary），每层标注 OS 类比
  - Block 之间有双向箭头（分页进出）
  - 底部：Agent 手持工具图标（标注 `core_memory_append` / `core_memory_replace`）
  - 右侧特性标签："平台原生" "OS 式分层" "显式编辑"
- **中间连接**：双向箭头，上方标注 "互补"，下方小字 "回答不同的问题"

**关键元素**：
- Mem0 侧的 API 箭头用虚线表示「外部调用延迟」
- Letta 侧的 Block 直接嵌入 Agent 运行时，无外部箭头
- 左栏底部标注 "54K stars"，右栏底部标注 "22K stars"
- 底部居中："选择取决于架构起点：已有框架 → Mem0；从零设计 → Letta"

**风格**：
- 扁平化架构图
- 类似系统架构文档中的组件图
- 组件用统一高度的矩形，保持视觉整齐

**比例**：16:9

**避免项**：
- 画得像一个「谁更好」的对比（必须是互补关系）
- 使用真实产品界面截图（用抽象组件图）
- 中间 "≠" 符号（用 "↔ 互补" 代替）

## 图 6：记忆方案决策树

**插入位置**：§6 实践建议 之后

**用途**：将 §6 的决策框架转化为可跟随的流程图，降低决策认知负荷

**画面构图**：
- 树状流程图，从上到下，菱形判断节点 + 圆角矩形终端节点
- **根节点（顶部）**："Agent 需要跨 session 记忆吗？"
  - **否** → 终端 A："框架内置方案足够"
    - 子选项：LangGraph state / CrewAI context / AutoGen Extensions
  - **是** → 判断节点 2："Agent 需要自我认知和主动记忆管理？"
    - **是** → 终端 B："Letta"
      - 标注："从零构建 | Block 抽象 | 精确 token 控制"
    - **否** → 判断节点 3："已有框架且需多 Agent 共享记忆？"
      - **是** → 终端 C："Mem0"
        - 标注："外挂集成 | 框架无关 | 多信号检索"
      - **否** → 判断节点 4："记忆语义与业务强耦合或有特殊合规？"
        - **是** → 终端 D："自建记忆系统"
          - 标注："HIPAA / 金融监管 / 审计策略嵌入"
        - **否** → 终端 E："Mem0（通用独立层）"
          - 标注："默认推荐 | 社区最大 | 54K stars"
- 终端节点用不同颜色区分：
  - 蓝色：框架内置
  - 青色：Letta
  - 深蓝：Mem0
  - 灰色：自建

**关键元素**：
- 判断节点用标准菱形，内部文字不超过 12 个字
- "是/否" 标注在箭头旁，箭头颜色跟随路径
- 终端节点下方标注社区规模（star 数）和核心特性（2-3 个关键词）
- 底部小字："决策树基于 2026-04 的框架状态，社区规模会变化"

**风格**：
- 扁平化流程图
- 类似 Mermaid / Draw.io 风格但更加精致
- 路径用颜色区分，保持树状清晰无交叉

**比例**：16:9（或 2:1 如果树太宽）

**避免项**：
- 分支超过 4 层（当前已控制在 4 层）
- 路径交叉（保持严格的从上到下树状）
- 终端节点没有明确标注 "为什么选这个"
