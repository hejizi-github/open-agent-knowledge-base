### 3.2 V3  phased batch pipeline（ADD-only 提取）

`add()` → `_add_to_vector_store()` 的 7 阶段流水线（来源：`mem0-memory-main.py:662-930`）：

| 阶段 | 操作 | 关键设计 |
|------|------|----------|
| Phase 0 | 上下文 gathering | 从 SQLite 取最近 10 条消息 |
| Phase 1 | 现有记忆检索 | 向量搜索 top_k=10，UUID→整数映射防幻觉 |
| Phase 2 | LLM 单通道提取 | `ADDITIVE_EXTRACTION_PROMPT`，一次 LLM 调用，JSON 输出 |
| Phase 3 | 批量 embedding | `embed_batch` 提取出的记忆文本 |
| Phase 4-5 | CPU 处理 + hash 去重 | MD5 哈希去重，跳过已存在的记忆 |
| Phase 6 | 批量持久化 | `vector_store.insert` 批量写入 |
| Phase 7 | 批量实体链接 | `extract_entities_batch` → 全局去重 → 批量嵌入 → 关联记忆 ID |

**ADD-only 语义**：提取出的记忆只有 `ADD` 事件，没有 `UPDATE`/`DELETE`。如果新消息与现有记忆语义重复，hash 去重会跳过；如果不重复，则作为新记录插入。旧记忆不会被修改或删除。

**历史记录**：所有操作写入 SQLite `history` 表，字段包括 `memory_id`, `old_memory`, `new_memory`, `event`, `created_at`, `is_deleted`。

## 4. 多信号检索（Multi-Signal Retrieval）

`search()` 方法的三路并行检索（来源：`mem0-readme.md` + `mem0-memory-main.py:1126-1440`）：

| 信号 | 机制 | 用途 |
|------|------|------|
| Semantic | 向量相似度搜索 | 捕获语义相关但词汇不同的记忆 |
| BM25 Keyword | 基于 lemmatized text 的关键词匹配 | 捕获精确术语匹配 |
| Entity Boost | 查询中的实体与 entity_store 关联记忆 | 提升同一实体相关记忆的排名 |

三路结果分别打分后融合（fusion），而非简单的串联或取交集。

## 5. 存储后端架构

### 5.1 向量存储（可插拔）

`pyproject.toml` 的 `vector_stores` extras 列出 15+ 后端：

- **默认/推荐**：Qdrant (`qdrant-client>=1.12.0`)
- **关系型+向量**：pgvector、PostgreSQL via `psycopg`
- **专用向量库**：Pinecone、Weaviate、Milvus、Chroma、Faiss、Elasticsearch、Redis/RedisVL、Upstash、Azure AI Search
- **NoSQL**：MongoDB (`pymongo`)、Databricks、Valkey

### 5.2 历史记录数据库

SQLite 固定用于历史记录（`history_db_path` 配置），与向量存储解耦。这意味着：
- 向量存储负责记忆的语义检索
- SQLite 负责记忆的审计追踪（who changed what when）

### 5.3 实体存储

懒加载的独立 collection（`{collection_name}_entities`），与主记忆存储共用同一向量后端，但独立 collection。实体存储的初始化会复用 Qdrant client 避免 RocksDB 锁竞争（来源：`mem0-memory-main.py:390-411`）。

## 8. 关键设计取舍

| 取舍 | Mem0 的选择 | 反事实 |
|------|------------|--------|
| 记忆更新策略 | ADD-only（只累积不覆盖） | 覆盖式更新会丢失历史版本，无法审计 |
| 提取策略 | 单通道 LLM 调用（无 agentic loop） | 多轮 agentic 提取延迟高，不适合实时场景 |
| 实体链接 | 懒加载独立 collection | 与主记忆同 collection 会污染检索信号 |
| 历史记录 | 固定 SQLite（非可插拔） | 统一存储后端会牺牲审计查询的灵活性 |
