# 继承/断裂矩阵分析方法

## 1. 方法定位

当分析两个技术项目的关系时（尤其是官方宣称"继任""升级""重构"关系的项目），技术社区常见的两种错误：
- **简单等同**：认为继任者 = 前身换皮（如"MAF = AutoGen 改名"）
- **简单否定**：认为继任者与前身完全无关（如"MAF 是全新项目"）

继承/断裂矩阵提供了一种结构化方法，强制区分"哪些是真的继承""哪些是断裂/重构"，从而得出更精确的关系判断。

## 2. 分析步骤

### Step 1：收集候选继承点

从以下维度系统扫描两个项目的相似性：

| 扫描维度 | 检查内容 | 证据来源 |
|---------|---------|---------|
| 概念命名 | 核心类/函数/配置项名称是否相同或高度相似 | 源码/API 文档 |
| 架构范式 | 底层设计模式是否一致（Actor/状态图/ReAct/角色驱动） | 架构文档/核心类设计 |
| 核心抽象 | 层数、包结构、模块边界是否相似 | 目录结构/导入路径 |
| 编排模式 | 预设的工作流/团队/聊天模式是否对应 | 官方示例/README |
| 协议支持 | 支持的通信/工具协议是否重叠 | 依赖列表/协议实现 |
| 生态策略 | 语言支持、商业平台绑定、License 策略 | GitHub API/pyproject.toml |
| 组织背景 | 发起方、维护团队、治理模式 | 官方博客/发布说明 |

**关键原则**：必须有源码级证据，不能仅凭 README 描述或社区传闻。

### Step 2：分类继承 vs 断裂

对每个候选点，使用以下判定标准：

**继承判定标准**（需同时满足 ≥2 条）：
1. 命名完全一致或高度相似（如 `MagenticOneGroupChat` → `MagenticBuilder`）
2. 源码实现中存在直接引用/兼容层/迁移指南
3. 官方文档明确承认继承关系
4. 核心算法/数据结构可追溯到前身项目

**断裂判定标准**（满足任意 1 条即可）：
1. 同一概念在前身和后身的实现方式完全不同（如包结构从 strict 三层 → 扁平 namespace）
2. 后身新增了前身完全没有的能力（如持久化、图工作流）
3. 后身移除了前身的关键能力（如 License 变更导致的使用限制变化）
4. 组织层面发生根本转移（研究院 → 产品团队）

### Step 3：构建矩阵

用双栏表格呈现：

```
| 继承方面 | 前身形态 | 后身形态 | 证据强度 |
|---------|---------|---------|---------|
| ... | ... | ... | 高/中/低 |

| 断裂方面 | 前身 | 后身 | 意义 |
|---------|------|------|------|
| ... | ... | ... | ... |
```

**证据强度分级**：
- **高**：源码可直接对比，或官方文档明确承认
- **中**：命名/概念相似，但实现细节不同
- **低**：仅 README 层面提及，无源码证据

### Step 4：得出关系结论

根据矩阵的填充结果，关系可分为四类：

| 类型 | 继承数 | 断裂数 | 典型特征 | 案例 |
|------|--------|--------|---------|------|
| **简单 rebranding** | 多 | 极少 | 换名不换药，源码可 diff | 极少见 |
| **产品化重构** | 中 | 多 | 继承概念，重新工程实现 | MAF vs AutoGen |
| **概念 fork** | 少 | 多 | 灵感来源，独立演化 | React vs Preact |
| **完全无关** | 极少 | N/A | 仅营销话术关联 | 需警惕假继承声明 |

## 3. 应用示例：MAF vs AutoGen

**继承点（5 处，证据强度均高）**：
1. 编排模式命名：`MagenticOneGroupChat` → `MagenticBuilder`，`SelectorGroupChat` → `GroupChatBuilder`
2. 双语言策略：Python + .NET 延续
3. Actor 模型运行时思想延续
4. MCP 协议支持延续
5. 核心概念命名（Agent/Tool/GroupChat）

**断裂点（8 处，证据强度均高）**：
1. 包结构：strict 三层 → 扁平 Tier 0/1/2 + namespace packages
2. 工作流：无原生图 → Graph-based + checkpoint + time-travel
3. 持久化：无 → DurableTask 原生
4. 协议：仅 MCP → MCP + A2A + AG-UI
5. UI：Studio（实验性）→ DevUI（生产级）
6. 代码执行：Docker → CodeAct + Hyperlight
7. 中间件：无概念 → 原生 Middleware
8. License：CC-BY-4.0 → MIT

**结论**：MAF 属于"产品化重构"——继承了概念和架构思想，但用完全不同的工程哲学重新实现。这不是简单 rebranding。

## 4. 常见陷阱

| 陷阱 | 表现 | 规避方法 |
|------|------|---------|
| README 陷阱 | 仅凭 README redirect 或"successor"声明判断 | 必须检查源码级差异 |
| 命名陷阱 | 概念名称相似 = 实现相同 | 检查 API 签名和内部实现 |
| 组织陷阱 | 同一公司 = 同一团队 = 同一产品 | 检查团队隶属（研究院 vs 产品部） |
| 时间陷阱 | 新项目发布 = 旧项目废弃 | 检查旧项目的维护状态声明 |
| 功能陷阱 | 新项目功能更多 = 完全重写 | 功能增量可能是新增而非替代 |

## 5. 可复用检查清单

- [ ] 是否从 ≥6 个维度扫描了候选继承点？
- [ ] 每个继承点是否有 ≥2 条证据支持？
- [ ] 每个断裂点是否有具体的"前身形态 vs 后身形态"对比？
- [ ] 是否检查了组织层面的转移（团队/部门/公司）？
- [ ] 结论是否属于四种关系类型之一（rebranding/产品化重构/概念 fork/完全无关）？
- [ ] 是否明确标注了每个断言的证据强度（高/中/低）？
